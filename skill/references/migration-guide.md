# 已有杂乱文档迁移指南

> 现有文档已经分散在十几个目录、分了"PRD/技术方案/接口/DB"几个分类，怎么渐进式迁移到 ai-team-docs 规范？

## 不要做的事

❌ **不要一次大搬迁**——把 100 篇老文档一夜之间挪到新目录。结果是：
- 链接全断（外部分享的 URL 失效）
- PR 评审看不懂
- 团队没接受新结构就被强迫使用，反弹大

❌ **不要立刻删老目录**——半年后还有人通过书签 / 群消息搜过去访问。

## 渐进式迁移（推荐）

### Phase 1：新文档按新规范写（最重要）

1. 先在 wiki 建好新结构（PRD Hub + 实现方案 wiki + features/reference/ops）
2. **从今天起**，所有新写的文档进新结构
3. 老文档**暂时不动**

效果：3-6 个月后，新文档自然占多数，老文档成为"历史档案"。

### Phase 2：高频文档重写

识别哪些老文档是"高频被引用"（搜索量高 / 在群里反复贴）：
- 核心架构图
- 数据模型主表
- 鉴权 / 部署流程

把这些**重写**到新结构（不是简单复制——按新模板的章节填）。老文档加一行：
```markdown
> ⚠️ 本文档已归档。最新版本：<link to new doc>
```

### Phase 3：低频文档保留为历史档案

- "v1 重构计划"、"3 年前的某次迁移记录" 这些低价值文档不要重写
- 但也不要删
- 在新结构的 `ops/历史归档` 或类似目录贴一个索引页，链回老地方
- 时间会自动让它们沉寂

### Phase 4：清理老目录（6-12 个月后）

当老文档真的没人访问（看 wiki 浏览统计）：
- 把还有少量价值的文档以 `[Deprecated]` 前缀挪到新结构
- 删除整个老目录（保留 git 历史 / wiki 历史，可恢复）

## 识别"反模式"先纠正

迁移前先看老文档有哪些反模式，规划重写时一并修：

### 反模式 1：单文档式 PRD + 技术方案
```
docs/Lucky 游戏/
├── 需求.md         ← PRD
├── 后端实现.md      ← 技术方案
└── 接口.md          ← 接口定义
```
→ 重写时：需求挪到 PRD Hub；后端实现 + 接口合并到 features/[Done] Lucky 游戏接入。

### 反模式 2：复制完整 schema 到 wiki
```markdown
# 用户表设计
| 字段 | 类型 |
|------|------|
| id | string |
| ... 50 行 ...
```
→ 重写时：业务含义 + 访问模式留下，字段表删掉，链回 repo model。

### 反模式 3：状态目录
```
docs/PRD/
├── 进行中/
├── 已发布/
└── 已归档/
```
→ 重写时：所有 PRD 进 PRD Hub flat；状态用标题前缀。

### 反模式 4：故障复盘藏在 feature 文档里
```
docs/Lucky 游戏接入/
└── 子页：2026-01 退款 bug 复盘
```
→ 重写时：复盘挪到 ops/故障复盘/。

## 沟通模板

迁移过程中，给团队发个通知：

> 📚 **文档治理升级**
>
> 从 X 月 X 日起，新写的 PRD / 技术方案进新结构：
> - PRD：[PRD Hub URL]
> - 后端实现方案：[Backend wiki URL]
> - 前端实现方案：[Frontend wiki URL]
>
> 老文档**暂不动**，但新文档不再写到老目录。
>
> 模板和规则见：[ai-team-docs README]
>
> 关键变化：
> 1. PRD 与技术方案分两个独立 wiki，不再混写
> 2. PRD 文档"实现追踪"表必须填，链各端实现方案
> 3. 实现方案文档头"对应 PRD"必填
> 4. wiki 涉及具体接口/字段时只写链接到 repo，不复制片段
>
> 有疑问找 @<owner>。

## 衡量迁移效果

3 个月后看：
- 新文档数 / 老文档数比例（应稳步上升）
- 团队成员是否知道"PRD 在哪 / 技术方案在哪"（随机问 5 个人）
- 一个 feature 的相关文档能否在 30 秒内找全（PRD → 技术方案 → 表设计 → ADR）

如果上述任一不达标，是规范没贯彻——回头看反模式纠正、模板优化、宣讲补强。

## 典型迁移时间线

| 时间 | 里程碑 |
|------|-------|
| Week 1 | 建新结构（PRD Hub + 实现方案 wiki），写好模板 |
| Week 2-4 | 第一批新文档进新结构（团队适应期） |
| Month 2-3 | 高频老文档重写到新结构 |
| Month 3-6 | 自然过渡，新结构成为默认 |
| Month 6-12 | 评估老结构清理 |

不要急。文档治理是马拉松。
