# 示例 ADR：多租户主键设计

> 示例 ADR Log 单条 —— 演示一个完整的 ADR 该写哪些字段。脱敏自真实项目。

---

## ADR-001：多租户主键设计模式（kolUserId#resourceId）

- **日期**：2025-08
- **状态**：已采纳
- **背景**：单租户表无法满足多 KOL 数据隔离需求；需要在不破坏 DynamoDB 分片性能的前提下实现租户级隔离
- **决策**：使用复合主键 `PK = "kolUserId#resourceId"` 实现多租户隔离，避免在查询里做租户过滤
- **替代方案**：
  - 每租户独立表（管理成本爆炸）
  - 共享表 + FilterExpression（性能差、成本高）
  - 独立 AWS 账号（与"租户隔离"粒度不匹配，跨租户查询难）
- **后果**：
  - 好：天然隔离、避开跨分区扫描、按需计费、实现简单
  - 坏：主键较长、跨租户查询需要额外 GSI 或扫描设计
- **重新评估触发条件**：业务进入"高频跨租户分析"场景；DynamoDB 的复合 PK 性能瓶颈出现
- **来源**：内部架构会议 2025-08；项目 architecture-issues.md 起源讨论

## 阅读指引

这是一个真实 ADR，不是模板。复制 [templates/ADR.md](../templates/ADR.md) 写新 ADR 时，参考它的字段填法：

✅ **好的填法**：
- 背景 1-3 句话讲清楚问题，不要写背景章节
- 决策 1-2 句话给结论，不要展开实现细节（实现细节在 features/ 文档里）
- 替代方案明确列出"被否决的"+ "原因"，不要只写"考虑过"
- 后果分"好"和"坏"两栏，不要只写好处

❌ **常见失误**：
- 把整个 PRD 抄进 ADR 的"背景"——ADR 是决策，不是需求文档
- 在 ADR 写代码 / SQL / API 字段——这些放对应的 features/ 或 tables/ 文档
- 多个决策塞一条 ADR——拆成 ADR-001 / ADR-002 / ...，每条独立追加
- 决策被废弃后改原文——保留原文，新写 ADR-N+1 标"已替代为 ADR-N+1"
