## 1. 背景（Why）

旧的"官方房（isOfficial）"二元模型把"运营归属"和"业务玩法"混在一个布尔字段里。每出一种特殊房间形态（神秘房、大神带飞），都要叠新枚举或新字段，产品扩展性差且语义不清。

## 2. 目标

- 把房间特性从"二元归属"改为**可组合的特性标签**，支持产品后续灵活叠加新玩法
- 优化推荐列表的内容质量（特殊房不再混入推荐流）

**非目标**：
- 不引入新的"特殊房间"运营玩法（v2.2 只做模型重构 + 现有两条产品线落地）
- 不做客户端兼容降级期（直接切换）

## 3. 使用场景

- **运营**：在管理端给某个房间一键标记"神秘房 🔮"或"大神带飞 ⭐"，可任意组合
- **C 端用户**：
  - 推荐流只看到普通房（特殊房不混入）
  - "更多"列表可按 `isMysteryRoom` / `isExpertRoom` 筛选
  - 详情页 / 列表显示对应类型徽章

## 4. 功能范围

### 4.1 核心
- 房间新增两个独立特性标志位：`isMysteryRoom`、`isExpertRoom`，可独立 + 任意组合（4 种状态）
- 推荐接口排除任一特性为 true 的房间
- "更多"接口支持按特性筛选
- 管理端：一键 toggle、批量编辑

### 4.2 状态组合
| 组合 | 显示 |
|------|------|
| 普通房 | 普通房 |
| 神秘房 | 🔮 神秘房 |
| 大神带飞 | ⭐ 大神带飞 |
| 神秘 + 大神 | 🔮 + ⭐ |

### 4.3 兼容性
- 历史房间默认 `false / false`（普通房）
- 旧字段 `isOfficial` / `roomOwnership` / `officialCount` / `personalCount` **直接移除**，不保留兼容期
- 管理端创建房间接口直接禁用（仅 C 端创建）

## 5. 数据 / 业务规则

- 一个房间可以同时是 神秘房 + 大神带飞
- 推荐流过滤规则：`isMysteryRoom == false AND isExpertRoom == false`
- "更多"接口筛选：参数缺省 = 全量；传 `true` / `false` = 精确筛选
