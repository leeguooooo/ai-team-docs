# 方法论总览

ai-team-docs 不是模板集合，是一套**协作约束**。模板只是这些约束的物化。

## 设计目标

1. **AI Agent 写文档不会越写越乱**——单一 feature 上下文集中在一处
2. **团队跨端协作不会双写漂移**——PRD/技术方案/接口/DB 各有归宿，相互链接
3. **文档不会和代码失同步**——所有"易过期内容"链回代码，不复制
4. **新人 30 秒就能找到该读什么**——目录扁平，状态可见

## 核心原则（5 条）

### 原则 1：按业务单元组织，不是按文档类型

详见 [why-ai-friendly.md](why-ai-friendly.md)

### 原则 2：PRD 与技术方案是不同维度

详见 [cross-team-linking.md](cross-team-linking.md)

### 原则 3：Source-of-Truth 必须显式

详见 [source-of-truth.md](source-of-truth.md)

### 原则 4：状态用标题前缀，不用目录

| 前缀 | 含义 | 适用 |
|------|------|------|
| `[Draft]` | 起草中 | PRD |
| `[Review]` | 评审中 | PRD |
| `[WIP]` | 进行中 | 通用 |
| `[Done]` | 已上线 | 通用 |
| `[Hold]` | 暂停 / 排期外 | 通用 |
| `[Deprecated]` | 已废弃 | 通用 |
| `[Cancelled]` | 已取消 | PRD |

**为什么不用目录管状态**：
- 移动文档会断链（wiki 节点 token / 文件路径变化）
- 状态频繁变化，移动成本不可接受
- 标题前缀可被 wiki/IDE 搜索过滤
- 历史 [Done] 是资产，不应"归档"到角落

### 原则 5：ADR Log 是追加式日志

每个架构决策一条独立条目，编号一旦分配不回收（包括废弃决策）：

```markdown
## ADR-001: <简短标题>
- **日期**：YYYY-MM
- **状态**：已采纳 / 已废弃 / 部分采纳 / 已设计待实施
- **背景**：1-3 句
- **决策**：1-2 句结论
- **替代方案**：列表（可选）
- **后果**：好的、坏的
- **重新评估触发条件**：（可选，"达到什么条件时回头看"）
- **来源**：repo 路径或会议
```

**为什么追加而不修改**：决策有历史，废弃也是决策。新决策应是 ADR-N+1，不是改 ADR-N。

## 小团队豁免

上面 5 条原则针对的是 **3+ 个端、3+ 人协作** 的中型团队场景。如果你是：

- 1-2 人的独立开发者
- 后端工程师兼 PM 角色
- 只有一个端（纯后端服务 / 纯 CLI 工具 / 纯 Web 前端）

**可以简化为单一 wiki**：
- 不分 PRD Hub 和实现方案 wiki，统一在一个项目 wiki 里
- `features/` 文档头部加个"业务背景"节代替 PRD（PM 视角）+ 后续技术方案（工程师视角）
- ADR / 表设计 / 术语表仍然分开维护（这些稳定参考资料的价值跨规模都成立）

**不能简化的部分**（即使一个人做也要遵守）：
- Source-of-truth 边界（代码 / 真相在 repo，决策在 wiki）
- 状态用标题前缀
- ADR 追加日志

## 反模式（不要这么做）

### ❌ 反模式 1：单一"项目文档"目录
```
docs/
├── PRD/
├── 设计/
├── 接口/
└── 部署/
```
理由：跨切的事情（一个 feature）必散落多处。

### ❌ 反模式 2：把代码细节复制到文档
```markdown
# 用户表设计

| 字段 | 类型 |
|------|------|
| id | string |
| name | string |
| ...50 行... |
```
理由：代码改了文档不知道。文档只写"业务含义 + 访问模式 + 链回 model 代码"。

### ❌ 反模式 3：PRD 写到接口字段级
PRD 应停在"业务规则 + 数据指标定义"，**接口契约和字段命名是技术方案的事**。否则 PM 改一行 PRD，三端工程师都要改方案。

### ❌ 反模式 4：用文档目录管状态
```
features/
├── 进行中/
├── 已发布/
└── 已归档/
```
理由：上线后要从"进行中"挪到"已发布"，链接全断。

### ❌ 反模式 5：故障复盘藏在某个 feature 里
```
features/[Done] 多租户迁移/
└── 子页：2026-01-数据隔离故障复盘
```
理由：一年后没人能找到。复盘在 `ops/故障复盘/`，独立时间线。

## 怎么开始

### 已有项目，想改造

1. 读 [why-ai-friendly.md](why-ai-friendly.md) 和 [source-of-truth.md](source-of-truth.md) 理解动机
2. 在 wiki / Notion 建两个空间：
   - PRD Hub（跨端共用）
   - <你的端> 实现方案（独立）
3. `npx ai-team-docs init` 把模板 scaffold 到本地（或参考 [templates/](../templates) 手动复制）
4. 选一个**正在做的 feature**，按新模板写一篇——验证模板和你的实际工作流匹配
5. 旧文档不要立刻搬，**新文档按新规范写**，老的自然消亡

### 新项目，从零开始

1. `npx ai-team-docs init --type all` 生成完整骨架
2. 配置 `~/.claude/skills/ai-team-docs/`（如果用 Claude Code）
3. 写第一篇 PRD（在 PRD Hub）
4. 后端建第一篇 [WIP] 技术方案，引用 PRD
