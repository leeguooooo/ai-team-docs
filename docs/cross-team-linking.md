# 跨团队双向链接协作

> 同一个需求 = **1 篇 PRD（PRD Hub）+ N 篇实现方案（各端 wiki）**。
> 文档之间用**显式链接**绑定。

## PRD 与技术方案的根本差异

| 维度 | PRD | 技术方案 |
|------|-----|---------|
| 作者 | PM | 工程师 |
| 范围 | 跨端：iOS / Android / Web / 后端 / 数据 | 单端实现细节 |
| 读者 | 全栈、PM、QA、运营、领导 | 该端工程师 |
| 修改频率 | 上线前定稿 | 实现迭代时频繁 |
| 修改权限 | PM 主导 | 各端 owner |
| 内容深度 | 业务规则、用户流程、数据指标 | API 契约、DB schema、状态机实现 |

**强行混在一起的代价**：
- iOS / 前端工程师不会进后端 wiki 找需求
- 同一需求改一次要在多个 wiki 同步
- PM 改 PRD 要进各端 wiki？工程师写实现要 PM 审批？
- 权限管理混乱

## 推荐架构

```
┌──────────────────────────────────────────────────────────┐
│  公司级：PRD Hub（独立 wiki 空间）                        │
│  ├── README                                              │
│  ├── _PRD 模板                                           │
│  └── PRD 文档（按业务线/时间，状态前缀管理）              │
└────────┬───────────┬───────────┬───────────┬─────────────┘
         │           │           │           │
         ↓           ↓           ↓           ↓
   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌─────────┐
   │ 后端 wiki│ │ Web wiki│ │ iOS wiki│ │ Android │
   │ features│ │ features│ │ features│ │  wiki   │
   └─────────┘ └─────────┘ └─────────┘ └─────────┘
        ↕           ↕           ↕           ↕
        └───────────┴───────────┴───────────┘
                双向链接：
                - PRD 文末"实现追踪"表 → 各端实现方案
                - 各端实现方案文档头 → 对应 PRD
```

## 双向链接规则（必须遵守）

### PRD 文档必须有"实现追踪"表

```markdown
## 7. 跨端范围 & 实现追踪

| 端 | 是否涉及 | 负责人 | 实现方案链接 | 状态 |
|----|---------|--------|------------|------|
| 后端 | ✅ | 张三 | [features/[WIP] xxx](URL) | [WIP] |
| Web | ✅ | 李四 | (待写) | - |
| iOS | ❌ | - | - | - |
| Android | ❌ | - | - | - |
| 数据 / 埋点 | ✅ | 王五 | [data/...](URL) | [WIP] |
```

### 实现方案文档必须有"对应 PRD"字段

```markdown
## 元信息
- **状态**：[WIP] / [Done] / [Hold] / [Deprecated]
- **对应 PRD**：[2026-Q2 Lucky 战绩排行](URL_TO_PRD)   ← 必填，没有 PRD 时填"无（内部技术债）"
- **负责人**：xxx
- **创建日期**：YYYY-MM-DD
```

### 链接形式

- 用平台原生链接（Lark wiki URL / Notion page link / Confluence link）
- 不要用相对路径（跨空间不通）
- 链接后写一句简短说明，避免链接腐烂时还能猜出指向

## 工作流（典型）

```
1. PM 写 PRD
   └── 各端的"实现追踪"暂时空着

2. 评审 → 排期
   └── PRD 状态从 [Draft] → [WIP]

3. 后端 / 前端 / iOS / Android 各自起稿实现方案
   ├── 各端在自己 wiki features/ 新建 [WIP] 文档
   └── 文档头部"对应 PRD"字段链回 PRD

4. PM 把各端实现方案链接补到 PRD 的"实现追踪"表

5. 上线
   ├── 各端实现方案改为 [Done]
   └── PRD 改为 [Done]，"实现追踪"标记完成
```

## 为什么不只在 PRD 一处写

技术上当然可以"PRD 写到很细，把每端的实现都包含"。但：

- **修改频率不同**：实现方案在开发期天天改，PRD 应稳定；混在一起 PM 看到的 PRD 永远是个"草稿"
- **读者期望不同**：领导读 PRD 想看业务，工程师读实现想看代码细节，混在一起两边都不爽
- **责任边界**：PRD 改坏了 PM 的锅，实现方案改坏了工程师的锅，混在一起责任模糊

## 一个 PRD 没有对应实现的情况

- **纯运营 / 数据需求**：仅涉及数据指标定义，没工程实现 → PRD "实现追踪" 表标 ❌ 即可
- **跨多端但本端不涉及**：iOS PRD 但 Android 不做 → Android 端打 ❌

## 一个实现方案没有对应 PRD 的情况

- **内部技术债重整**（如多租户迁移）→ 在"对应 PRD"字段写"无（内部技术债务整治）"
- **bug 修复 / 重构**：太小，不写实现方案，commit message 即可
- **大重构**：写实现方案，"对应 PRD"字段写"无（架构演进）"

不能因为"没 PRD"就不写实现方案；也不能因为"没实现"就不写 PRD（运营需求例外）。

## 反模式

### ❌ 单文档式（PRD + 技术方案合写）
PM 和工程师抢同一篇 doc，git history 像战场。

### ❌ 把后端实现方案放在 PRD Hub
后端工程师改实现方案要进 PRD Hub？跨空间权限混乱。

### ❌ 没有"对应 PRD"字段
半年后没人知道这个 [Done] 实现到底在解决什么。

### ❌ "实现追踪" 表只是装饰
PRD 文末有表但从不更新——和没有一样。
