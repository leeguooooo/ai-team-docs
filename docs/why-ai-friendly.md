# 为什么 AI 时代要用"自包含 + 业务单元"结构

> 真正的核心原则是**单文档自包含**，扁平只是它的常见副产物。下文讲清楚为什么。

## 传统结构（按文档类型切）的问题

```
project-docs/
├── 01-产品需求 PRD/
├── 02-技术方案/
├── 03-接口文档/
├── 04-数据库设计/
└── 05-运维部署/
```

这是**前 AI 时代**的结构——按文档类型分目录，看起来清爽。

但放到 AI Agent 协作里就崩了：

### 问题 1：一个改动散落 4 处

做"Lucky 游戏接入"这个 feature，需要：
- 在 `01-PRD/` 写需求
- 在 `02-技术方案/` 写方案
- 在 `03-接口文档/` 写接口
- 在 `04-数据库/` 写表设计

**AI 写文档时要在 4 处同步**——改方案多半忘了改 PRD，加字段忘了改接口文档。半年后这 4 篇必然漂移。

### 问题 2：AI 检索时拼不出全貌

让 Claude 回答"Lucky 游戏怎么集成的？"——它得：
1. 搜 `01-PRD/` 找到背景
2. 搜 `02-技术方案/` 找到方案
3. 搜 `03-接口文档/` 找接口
4. 搜 `04-数据库/` 找 schema

每次都不完整。**AI 友好的内容布局应该让一篇文档自含上下文**。

### 问题 3：状态目录是反模式

`01-PRD/` 下还分 `进行中/已发布/归档/`？上线一次要在三个目录间挪文档，wiki 节点 token 变，链接全断。

## AI 友好结构（按业务单元切）

```
project-docs/
├── features/                    ← 主战场
│   ├── _模板
│   ├── [WIP] 2026-Q2 Lucky 游戏战绩
│   ├── [Done] Lucky 游戏接入
│   ├── [Done] 房间分类系统 v2.2
│   └── ...
├── reference/                   ← 跨 feature 稳定参考
│   ├── 术语表
│   ├── 架构总览
│   ├── ADR Log
│   ├── API 总览（链接 repo）
│   └── 数据模型索引 + tables/
└── ops/                         ← 运维实战
    ├── 故障复盘
    └── 发布记录
```

**关键变化**：
- **一个 feature = 一篇自包含文档**：背景 + 方案 + API 契约 + DB schema + 风险 + 上线，都在一篇
- **状态用标题前缀**：`[WIP] / [Done] / [Hold] / [Deprecated]`，不用目录
- **reference 只放跨 feature 的稳定知识**：架构、ADR、术语、表索引，少而精
- **ops 单独立**：复盘和发布记录有自己的时间线

## 对比

| 维度 | 传统按类型 | AI 友好按单元 |
|------|-----------|--------------|
| 一个 feature 文档 | 散落 4 处 | 1 篇自包含 |
| AI 写文档 | 4 处同步 | 1 篇填空 |
| AI 查上下文 | 需跨多目录拼 | 单文档完整 |
| 状态管理 | 移动目录（断链）| 改标题前缀 |
| 一级目录数 | 6+ | 3（features/reference/ops）|
| 模板位置 | 散在各目录 | features/ 根有一份 |

## 但单文档会膨胀，怎么办？

这是常见反驳。我们的回答：

### feature 文档保持薄

不在 feature 文档里复制完整 schema 或 API 字段。**链接到 repo 真相**：

```markdown
## 4. API 契约
新增 endpoint：
| Method | Path | 说明 |
|--------|------|------|
| POST | /api/lucky-game/order/create | 创建订单 |

详细字段见 `api/openapi.yaml` 中对应 schema。
```

### 跨 feature 的稳定信息抽到 reference

数据模型变化频繁的话，`reference/数据模型索引` 不写完整字段，而是导航到 `tables/<表名>` 子页：

```
reference/
├── 数据模型索引       ← 表清单 + 全局规则
└── tables/            ← 一表一页，详细访问模式
    ├── rooms-v2
    ├── badges-v3
    ├── tags-v3
    └── lucky-game 系列表
```

每张表的字段定义仍然链回 repo `internal/models/<表>.go`。

## 怎么衡量结构是不是"AI 友好"

3 个测试：

1. **单文档完整度**：随便挑一篇 feature，让 Claude 不读其他文档，能否解释这个 feature？
2. **状态可见性**：在 wiki 列表里 30 秒内能看出"现在在做什么 / 已上线什么 / 哪个是历史"？
3. **跨切跳转成本**：从 PRD 跳到对应实现、从实现跳到对应表、从表跳到对应 ADR——每一跳都该是一次点击。

如果 3 个测试都通过，结构就 AI 友好。

## 关于"扁平"的边界

**扁平是手段，不是目的**。真正的核心原则是单文档自包含 + 跨切用链接而不是复制。

具体推论：
- **小型项目（< 30 个 feature）**：直接扁平 `features/[WIP] xxx.md`，AI 一次性看全
- **中型项目（30-100 个 feature）**：在 `features/` 下按业务线分一层子目录（如 `features/auth/[WIP] xxx`、`features/billing/[Done] yyy`），保持每个子目录里仍然扁平
- **大型项目（100+ 个 feature）**：业务线层级 + 子项目级 wiki 空间（每个产品线一个空间）

**不要因为追求"扁平"而把 200 个 feature 堆在一个目录里**——那不是 AI 友好，是上下文窗口爆炸。也不要因为"按部门分目录看起来整洁"而把一个 feature 拆到 4 个目录里——那是反 AI。

判断标准始终是：**AI 读一个 feature 能不能在合理上下文里看全**。
